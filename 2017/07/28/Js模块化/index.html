让我们再回忆一下那个打开网站页面广告飘来飘去的时代，但在当时看来，这也许是一件极为正常的事情。

那个时候并没有前端工程师，页面没有这时候这么美观，需求也没有这么复杂，工程师在页面上随便写写js就能搞定需求。

如果你用原生js写过项目，你一定会遇到这些问题：

  1. 尴尬的全局变量
    Js出了名的弱语言类型，变量作用域闭包原型链问题也成为面试笔试部分的一道靓丽的风景线。Js只有函数作用域，没有块级作用域，即使是if/for等有{}的结构体，也并不能具备自己的作用域。

  2. 函数命名冲突
    唔，本身起名字也是一件稍费神的事情。如果你刚接手了一个项目，小王同学已经写了一半了，此时你进入项目，很巧的你想要实现一个跟小王同学之前写过的一个类似的功能函数，你可能会十分担心自己的函数名字是否会跟小王同学起的函数名字冲突，只能用一种丑陋的方式进行（偷偷在函数名字后面加个1）。

  3. 依赖关系
    自上而下的执行方式使得如果b.js依赖a.js，引入a.js的<script>标签就必须写在b.js前面

CommonJs

Node.js的出世将JavaScript带到了服务器端，社区大神发力制定了一系列规范，叫做CommonJs。

CommonJS模块的代码都运行在模块作用域，不会污染全局作用域，模块可以多次加载，用 require 引入模块，exports 向外暴露api。

它主要是为服务器端JavaScript运行库设计的，模块是同步加载的，这使得难以在浏览器中运行CommonJS代码。

为了达成浏览器端模块化开发的目的，还有很多的规范随着技术的产生推广而被知道。👇

AMD

AMD（异步模块定义）是 RequireJS 在推广过程中对模块定义的规范化产出。
